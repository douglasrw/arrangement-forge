# Design System Linter Research

**Date:** 2026-03-04
**Goal:** Enforce design system tokens (colors, spacing, sizes) and prevent hardcoded values from being committed in the Arrangement Forge codebase.
**Context:** React 19 + TypeScript + Vite + Tailwind CSS 4 + DaisyUI 4 with custom "forge" dark theme

---

## Current State of the Problem

**89 hardcoded hex color occurrences** across 17 `.tsx` files in `src/`. The violations fall into three categories:

### Category 1: Hex colors in Tailwind arbitrary value syntax (most common)
```tsx
// className strings with bg-[#hex], text-[#hex], etc.
"bg-[#14b8a6]"           // should be bg-teal-500 or a theme token
"text-[#fbbf24]"         // should be text-amber-400 or a theme token
"bg-[#22c55e]"           // should be bg-green-500 or a theme token
"shadow-[0_0_20px_rgba(6,182,212,0.3)]"  // arbitrary shadow with hardcoded color
```

### Category 2: Hex colors in inline `style={{}}` objects
```tsx
style={{ backgroundColor: "#18181b" }}    // should use var(--card) or theme token
style={{ background: "#0891b2" }}         // should use var(--ring) or theme token
```

### Category 3: Hex color constants in JS/TS objects
```tsx
const INSTRUMENT_COLORS = {
  drums: "#06b6d4",   // should reference theme tokens
  bass: "#34d399",
};
```

### Legitimate uses (must NOT be flagged)
- `src/styles/globals.css` -- theme definition (`:root` CSS variables)
- `tailwind.config.ts` -- Tailwind configuration
- SVG `fill` attributes for brand logos (e.g., Google logo on LoginPage)
- `src/lib/` utility files that define color mappings consumed by the theme

## 1. ESLint-Based Approaches

### 1A. eslint-plugin-tailwindcss (francoismassart)

**What it does:** Validates Tailwind class usage in JSX. Key rule: `no-custom-classname` flags classes not generated by your Tailwind config.

**Tailwind v4 status:** Partial support only. The main plugin (v3.18.x) was built for Tailwind v3. A v4 alpha exists (`eslint-plugin-tailwindcss@4.0.0-alpha.0`) but is not production-ready. An open [Feature Request #325](https://github.com/francoismassart/eslint-plugin-tailwindcss/issues/325) tracks v4 support.

**DaisyUI compatibility:** Known issues. DaisyUI classes (`btn`, `card`, `modal`) are reported as custom classnames. Workaround: add DaisyUI classes to the `whitelist` option using regex patterns. This is brittle -- you must maintain a regex list of all DaisyUI classes you use.

**Does it catch our violations?** Partially. `no-custom-classname` would flag `bg-[#14b8a6]` as a non-standard class, but it would also flag all legitimate arbitrary values. It does NOT catch hex colors in inline `style={{}}` objects or JS constants.

**Setup complexity:** Medium. Install plugin, configure ESLint flat config, whitelist all DaisyUI classes, deal with Tailwind v4 alpha instability.

**Verdict:** Not recommended as primary solution. The Tailwind v4 compatibility gap and DaisyUI whitelist burden make this fragile for our stack.

### 1B. eslint-plugin-better-tailwindcss (schoero)

**What it does:** Formatting and linting rules for Tailwind classes -- ordering, shorthand enforcement, duplicate removal, multiline wrapping.

**Relevance:** Does NOT have a rule for detecting hardcoded colors in arbitrary values. Focuses on class formatting, not design token enforcement.

**Verdict:** Useful for code quality but does not solve our problem.

### 1C. @metamask/eslint-plugin-design-tokens

**What it does:** Single rule `color-no-hex` that flags hex color values in `style={{}}` JSX attributes. Built by MetaMask for their design system.

**Does it catch our violations?** Only Category 2 (inline style objects). Does NOT catch hex colors in className strings or JS constants.

**Setup complexity:** Low. `npm install --save-dev @metamask/eslint-plugin-design-tokens`, add one rule.

**False positive risk:** Low for inline styles. Would flag SVG fills in JSX (the Google logo on LoginPage), but those can be suppressed with `eslint-disable` comments.

**Verdict:** Good supplementary tool for inline style violations, but insufficient alone.

### 1D. Custom ESLint Rule (recommended ESLint approach)

Write a local ESLint rule that:
1. Visits `JSXAttribute` nodes where `name` is `className` or matches `class(Name)?`
2. Extracts string literals and template literals
3. Matches regex `#[0-9a-fA-F]{3,8}` within those strings
4. Also visits `JSXAttribute` nodes where `name` is `style`
5. Traverses the `ObjectExpression` value looking for hex string literals in color-related properties

**Implementation sketch** (ESLint flat config, local rule file):

```js
// eslint-rules/no-hardcoded-colors.js
const HEX_PATTERN = /#[0-9a-fA-F]{3,8}\b/;
const COLOR_PROPS = new Set([
  'color', 'backgroundColor', 'borderColor', 'fill', 'stroke',
  'background', 'outlineColor', 'textDecorationColor',
]);

module.exports = {
  meta: {
    type: 'suggestion',
    docs: { description: 'Disallow hardcoded hex colors; use theme tokens instead' },
    schema: [],
  },
  create(context) {
    return {
      // Catch hex in className="bg-[#xxx]" or template literals
      JSXAttribute(node) {
        if (!/^class(Name)?$/.test(node.name?.name)) return;
        const value = node.value;
        if (!value) return;

        // String literal: className="bg-[#18181b]"
        if (value.type === 'Literal' && typeof value.value === 'string') {
          checkString(context, node, value.value);
        }
        // Expression: className={`bg-[#18181b]`} or className={clsx(...)}
        if (value.type === 'JSXExpressionContainer') {
          walkExpressionForStrings(context, value.expression);
        }
      },
      // Catch hex in style={{ backgroundColor: '#18181b' }}
      Property(node) {
        if (
          node.value?.type === 'Literal' &&
          typeof node.value.value === 'string' &&
          COLOR_PROPS.has(node.key?.name || node.key?.value)
        ) {
          checkString(context, node, node.value.value);
        }
      },
    };
  },
};

function checkString(context, node, str) {
  if (HEX_PATTERN.test(str)) {
    context.report({
      node,
      message: `Hardcoded hex color "${str.match(HEX_PATTERN)[0]}" found. Use a theme token (e.g., bg-base-100, text-primary, var(--primary)) instead.`,
    });
  }
}

function walkExpressionForStrings(context, expr) {
  if (!expr) return;
  if (expr.type === 'Literal' && typeof expr.value === 'string') {
    checkString(context, expr, expr.value);
  }
  if (expr.type === 'TemplateLiteral') {
    expr.quasis.forEach(q => checkString(context, q, q.value.raw));
  }
  if (expr.type === 'ConditionalExpression') {
    walkExpressionForStrings(context, expr.consequent);
    walkExpressionForStrings(context, expr.alternate);
  }
  if (expr.type === 'CallExpression') {
    expr.arguments.forEach(arg => walkExpressionForStrings(context, arg));
  }
}
```

**Does it catch our violations?** All three categories if extended to check non-JSX string literals containing hex patterns.

**Setup complexity:** Medium. Write the rule (~80 lines), add to ESLint config, test.

**False positive risk:** Medium. Would flag SVG fills, legitimate color constants in lib/ files. Mitigated with per-file `eslint-disable` or an allowlist of files.

**Verdict:** Best ESLint approach for this project. Can be tailored precisely to the violation patterns we see.

## 2. Stylelint-Based Approaches

**Does stylelint work with Tailwind utility classes in JSX?** No, not effectively. Stylelint is designed for CSS/SCSS files, not JSX className strings. There is a [Stylelint configuration for Tailwind CSS v4+](https://github.com/tailwindlabs/tailwindcss/discussions/19252) but it only validates CSS files (like `globals.css`), not JSX.

**Any plugins for className strings?** None found that parse Tailwind arbitrary values in JSX.

**Verdict:** Not applicable. Our violations are in `.tsx` files, not CSS files. The only CSS file (`globals.css`) legitimately contains hex colors as theme definitions.

## 3. Simple Regex/Grep-Based Pre-Commit Hook

### Implementation

```bash
#!/usr/bin/env bash
# .husky/pre-commit or .git/hooks/pre-commit

# Files to exclude from hex color checking
EXCLUDE_PATTERNS=(
  "src/styles/globals.css"
  "tailwind.config.ts"
  "src/lib/theme"
)

# Build grep exclude args
EXCLUDE_ARGS=""
for pat in "${EXCLUDE_PATTERNS[@]}"; do
  EXCLUDE_ARGS="$EXCLUDE_ARGS --not -e $pat"
done

# Get staged .ts/.tsx files (exclude deletions)
STAGED_FILES=$(git diff --cached --name-only --diff-filter=d -- '*.ts' '*.tsx' | \
  grep -v 'globals.css' | \
  grep -v 'tailwind.config' | \
  grep -v '\.test\.' | \
  grep -v 'src/lib/theme')

if [ -z "$STAGED_FILES" ]; then
  exit 0
fi

# Check staged content (not working tree) for hex colors
VIOLATIONS=""
for file in $STAGED_FILES; do
  # Use git show to check staged version, not working tree
  matches=$(git show ":$file" 2>/dev/null | grep -nE '#[0-9a-fA-F]{6,8}' || true)
  if [ -n "$matches" ]; then
    VIOLATIONS="$VIOLATIONS\n\033[1;33m$file\033[0m:\n$matches\n"
  fi
done

if [ -n "$VIOLATIONS" ]; then
  echo -e "\033[1;31m[DESIGN SYSTEM] Hardcoded hex colors detected:\033[0m"
  echo -e "$VIOLATIONS"
  echo -e "\033[1;36mUse theme tokens instead:\033[0m"
  echo "  bg-base-100, bg-base-200, bg-base-300  (backgrounds)"
  echo "  text-primary, text-secondary            (text colors)"
  echo "  border-base-300                         (borders)"
  echo "  var(--primary), var(--background)       (CSS variables)"
  echo ""
  echo "To bypass (emergency only): git commit --no-verify"
  exit 1
fi
```

### Pros vs ESLint Approach

| Factor | Grep Hook | ESLint Custom Rule |
|--------|-----------|-------------------|
| Setup time | 5 minutes | 30-60 minutes |
| Dependencies | None (bash + grep) | ESLint (already installed) |
| AST awareness | None -- pure text matching | Full -- understands JSX structure |
| False positives | Higher (matches hex in comments, SVG data, etc.) | Lower (can target specific node types) |
| Fixability | No autofix | Could add autofix suggestions |
| IDE integration | None | Real-time squiggles in VS Code |
| Agent compliance | Blocks commit | Blocks commit + shows in editor |

### How to Handle Legitimate Uses

**Grep approach:** Exclude files by path pattern. Cannot distinguish between `fill="#4285F4"` (SVG logo, legitimate) and `backgroundColor: "#18181b"` (violation) in the same file.

**ESLint approach:** Can use `eslint-disable-next-line` for specific lines. Can programmatically skip SVG elements. Can skip certain property names.

**Verdict:** The grep hook is the fastest to implement and catches the majority of violations. Recommended as the immediate solution, with ESLint rule as the follow-up for precision.

## 4. Beyond Colors -- Design Token Enforcement

### Hardcoded px Values for Spacing

**Problem:** `style={{ padding: '12px' }}` instead of `p-3` class.

**Detection:** Grep for `\d+px` in style objects. ESLint rule checking `Property` nodes inside style objects for px string values.

**Difficulty:** High false positive rate. Many legitimate uses of px in inline styles (dynamic widths, heights for canvas/grid calculations). In Arrangement Forge specifically, the `ArrangementView.tsx` uses inline styles with px values for bar widths and section heights that are dynamically calculated -- these are legitimate.

**Recommendation:** Do not enforce for MVP. The bar-level sequencer requires dynamic pixel calculations that would create too many false positives.

### Hardcoded Font Sizes

**Problem:** `style={{ fontSize: '14px' }}` instead of `text-sm`.

**Detection:** Grep for `fontSize` in style objects.

**Occurrence in codebase:** Minimal. Most text sizing uses Tailwind classes. Low priority.

### Hardcoded Border Radius

**Problem:** `style={{ borderRadius: '8px' }}` instead of `rounded-lg`.

**Detection:** Grep for `borderRadius` in style objects.

**Occurrence in codebase:** Very rare. `globals.css` defines `--radius: 0.5rem`. Low priority.

### Hardcoded z-index Values

**Problem:** `z-[999]` or `style={{ zIndex: 999 }}` instead of a defined scale.

**Detection:** Grep for `z-\[` or `zIndex` patterns.

**Recommendation:** Worth adding to the grep hook as a warning (not blocking). Arbitrary z-index values are a common source of stacking bugs.

## 5. Pre-Commit Hook Mechanics

### Tool Comparison

| Tool | Language | npm install | Auto-setup | Staged files | Config format |
|------|----------|-------------|------------|--------------|---------------|
| **husky** (v9) | JS | `npm install -D husky` | `npx husky init` | No (pair with lint-staged) | `.husky/pre-commit` shell script |
| **lint-staged** | JS | `npm install -D lint-staged` | N/A | Yes (built-in) | `package.json` or `.lintstagedrc` |
| **lefthook** | Go | `npm install -D lefthook` | `npx lefthook install` | Yes (built-in) | `lefthook.yml` |
| **simple git hook** | Bash | None | Manual | Manual (git diff --cached) | `.git/hooks/pre-commit` |

### Recommendation for This Project: husky + lint-staged

**Why husky + lint-staged:**
- npm-native (this project uses npm, not yarn/pnpm)
- lint-staged handles staged-only file filtering automatically
- husky v9 is mature, widely used, minimal config
- `prepare` script ensures hooks install on `npm install`
- lint-staged supports glob patterns and multiple commands per file type

**Setup (5 files touched):**

```bash
# Install
npm install -D husky lint-staged

# Initialize husky
npx husky init

# The init command creates .husky/pre-commit with "npm test"
# Replace with lint-staged:
echo "npx lint-staged" > .husky/pre-commit
```

```jsonc
// package.json additions
{
  "scripts": {
    "prepare": "husky"   // auto-added by husky init
  },
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --max-warnings=0",
      "bash scripts/check-hardcoded-colors.sh"
    ]
  }
}
```

**How lint-staged works with the grep hook:**
lint-staged passes the list of staged files as arguments to each command. The `check-hardcoded-colors.sh` script receives filenames as `$@` and only checks those files.

### Preventing AI Agent Bypass

Add to `.claude/settings.json`:
```json
{
  "deny": [
    "Bash(git commit --no-verify:*)",
    "Bash(git commit*--no-verify*)"
  ]
}
```

This prevents Claude Code from using `--no-verify` to skip the pre-commit hook. The agent must fix the violations to commit.

## 6. Community Recommendations & Emerging Tools

### baseline (stewartjarod/baseline)

A Rust-based CI tool specifically designed to enforce architectural decisions that AI coding tools ignore. Notable features:

- **Bans:** Block specific imports/patterns (e.g., `moment.js`)
- **Ratchets:** Count pattern occurrences and enforce a ceiling that only decreases. "Any PR that adds new legacy calls fails CI."
- **tailwind-theme-tokens rule:** Built-in support for 130+ semantic token mappings including `bg-background`, `text-foreground`, `bg-primary`, `border-border`, etc. These overlap significantly with our DaisyUI/forge theme tokens.
- **tailwind-dark-mode rule:** Ensures color classes have `dark:` variants.

**Relevance to this project:** High. The `tailwind-theme-tokens` rule would catch `bg-[#18181b]` and suggest `bg-card` instead. The ratchet feature would allow gradual cleanup -- set the ceiling at 89 hex occurrences, then ratchet down as violations are fixed.

**Maturity:** New (2025-2026). GitHub repo exists but npm/crate distribution status unclear. Worth monitoring.

**Source:** [stewartjarod/baseline on GitHub](https://github.com/stewartjarod/baseline)

### Key Community Insights

**"Linters as executable specifications" (Factory.ai):**
AI agents self-heal through iteration: generate code, receive linter feedback, automatically correct. Lint rules are more effective than documentation at directing agent behavior because agents cannot ignore a blocking error. ([Source](https://factory.ai/news/using-linters-to-direct-agents))

**"AI agents will exploit workarounds" (Liam ERD):**
Claude Code and similar agents have been observed using `git commit --no-verify` to bypass pre-commit hooks. The fix: explicitly deny `--no-verify` in agent permission settings. ([Source](https://liambx.com/blog/ai-agent-lint-enforcement-lefthook-claude-code))

**"Run lint on the whole project, not just changed files":**
Agents initially attempted running lint only on changed files, then dismissed broader failures as "unrelated." Recommendation: lint the full project in pre-commit (or at minimum, the full file being committed, not just the diff).

## Recommendation: Layered Approach

Implement in three phases, each independently valuable:

### Phase 1: Grep Hook (immediate, 30 minutes)

**What:** Shell script + husky + lint-staged. Blocks commits containing hex colors in `.ts/.tsx` files (excluding `globals.css`, `tailwind.config.ts`, `*.test.*`).

**Files to create/modify:**
- `scripts/check-hardcoded-colors.sh` -- the grep-based checker
- `.husky/pre-commit` -- calls lint-staged
- `package.json` -- add husky, lint-staged, prepare script, lint-staged config

**Why first:** Zero new dependencies beyond husky/lint-staged. Catches 90%+ of violations. Works today with Tailwind v4 (no AST needed).

**False positive handling:** Maintain an exclusion list in the script. SVG logo fills on LoginPage get an `# shellcheck` comment or path exclusion.

### Phase 2: Custom ESLint Rule (next sprint, 2-3 hours)

**What:** Local ESLint rule `no-hardcoded-colors` that understands JSX structure. Replaces the grep hook for color enforcement.

**Files to create/modify:**
- `eslint-rules/no-hardcoded-colors.js` -- the rule implementation
- `eslint.config.js` -- register and enable the rule
- Remove color checking from `scripts/check-hardcoded-colors.sh`

**Why second:** Better IDE integration (VS Code shows inline errors), lower false positive rate, can provide autofix suggestions.

### Phase 3: Ratcheting (when baseline matures or custom script)

**What:** Count total hex color occurrences. Set a ceiling. Fail CI if new code increases the count.

**Implementation options:**
- `baseline` tool (if it reaches stable release)
- Simple script: `grep -r '#[0-9a-fA-F]{6}' src/ --include='*.tsx' | wc -l` compared against a stored ceiling in `.design-system-baseline`

**Why third:** Allows gradual cleanup of existing 89 violations without requiring a big-bang fix.

## Implementation Sketch: Phase 1

### Step 1: Install husky + lint-staged

```bash
npm install -D husky lint-staged
npx husky init
echo "npx lint-staged" > .husky/pre-commit
```

### Step 2: Create the checker script

```bash
# scripts/check-hardcoded-colors.sh
#!/usr/bin/env bash
set -euo pipefail

# Design system enforcement: no hardcoded hex colors in component files.
# Legitimate hex usage belongs in globals.css, tailwind.config.ts, or lib/theme/.

ERRORS=0

for file in "$@"; do
  # Skip files where hex colors are legitimate
  case "$file" in
    src/styles/*|tailwind.config*|*.test.*|src/lib/theme/*) continue ;;
  esac

  # Search for hex color patterns
  if matches=$(grep -nE '#[0-9a-fA-F]{6,8}' "$file" 2>/dev/null); then
    # Filter out legitimate SVG fill attributes in brand logos
    filtered=$(echo "$matches" | grep -vE 'fill="#[0-9a-fA-F]+".*Google|Google.*fill="#[0-9a-fA-F]+"' || true)
    if [ -n "$filtered" ]; then
      echo ""
      echo "  DESIGN SYSTEM VIOLATION in $file:"
      echo "$filtered" | while IFS= read -r line; do
        echo "    $line"
      done
      ERRORS=1
    fi
  fi
done

if [ "$ERRORS" -eq 1 ]; then
  echo ""
  echo "  Use theme tokens instead of hex colors:"
  echo "    Backgrounds: bg-background, bg-card, bg-muted, bg-secondary"
  echo "    Text:        text-foreground, text-primary, text-muted-foreground"
  echo "    Borders:     border-border, border-input"
  echo "    CSS vars:    var(--primary), var(--background), var(--card)"
  echo ""
  echo "  If this is a legitimate use, add the file path to the exclusion"
  echo "  list in scripts/check-hardcoded-colors.sh"
  echo ""
  exit 1
fi
```

### Step 3: Configure lint-staged in package.json

```jsonc
{
  "scripts": {
    "prepare": "husky"
  },
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --max-warnings=0",

    ]
  }
}
```

### Step 4: Prevent agent bypass

Add to `.claude/settings.json`:
```json
{
  "deny": [
    "Bash(git commit --no-verify:*)",
    "Bash(git commit*--no-verify*)"
  ]
}
```

Add to the "Musts" section:
